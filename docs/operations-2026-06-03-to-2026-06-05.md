# Operations Summary, 2026-06-03 to 2026-06-05

This is a sanitized audit of the local changes made while stabilizing the Codex remote GCP path.

## What Changed

- Split the menu app actions into SSH refresh and one-shot Remote GCP repair.
- Shortened the menu-bar title to `KC` to reduce menu-bar crowding.
- Added a stay-awake LaunchAgent and menu controls.
- Added a narrow sudo route helper for repairing only the GCP host route.
- Added detection for stale VPN/`utun*` routing to the GCP host.
- Added local SOCKS and HTTP proxy health checks.
- Added remote `10800` listener and data-plane checks.
- Added cleanup for stale SSH ControlMaster state.
- Made Remote GCP repair a single forced cleanup action that resets local `1080/7890`, remote `10800`, stale non-tty SSH sessions, and remote Codex app-server processes before rebuilding and validating the path.
- Removed full `codex exec openai-ok` as the default fast validation because model/tool errors can be mistaken for network failure.
- Tested an experimental HTTP-to-SOCKS bridge; it is kept in `tools/` but is not the default path.

## What Was Tested

- `curl` through local SOCKS `1080`.
- `curl` through local HTTP `7890`.
- `curl` from the remote workspace through remote HTTP `10800`.
- ChatGPT Codex endpoint reachability through `10800`.
- Direct Codex CLI validation through the remote wrapper.
- Route switching when moving between company Wi-Fi, home Wi-Fi, phone hotspot, and Feilian/VPN states.

## Cost-Relevant Finding

The GCP VM was not CPU-bound during the latest check; load was near zero. The cost risk is traffic volume. The VM had hundreds of GB of cumulative network traffic since boot, and the local SOCKS process had many active client connections. That is consistent with Codex app-server traffic and reconnect loops going through the GCP tunnel.
